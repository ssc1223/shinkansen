# Shinkansen 效能報告

> 測試版本：v1.9.16 ｜ 測試日期：2026-05-14
> 上次測試：v1.0.23 / 2026-04-11（差異對比見文末）

## 結論

Shinkansen 翻譯整頁維基百科（接近一千段文字），總耗時約 48 秒、費用 $0.08 美元，DOM 節點數反而下降 3,171 個（中文比英文精簡）。整體成本結構與上次測試（v1.0.23）幾乎一致；翻譯期間 95%+ 的時間花在等 API，本地 CPU 與 DOM 影響可忽略。

## 白話文摘要

拿一個非常長的英文維基百科頁面（Taiwan 條目，989 段文字、超過 21 萬個字元）做壓力測試，結果如下：

**費用幾乎沒變。** 用最便宜的 Flash Lite 模型翻譯這整個千段大頁面，API 費用 $0.077 美元（約台幣 2.4 元），跟一年前 v1.0.23 測得的 $0.078 美元幾乎完全一致——代表這一年間多出來的功能（顯式雙語、SPA 自動翻譯、術語表、中國用語黑名單、YouTube 字幕等）對「網頁翻譯」這條路徑的 token 開銷沒有實質影響。

**網頁不會變卡。** Extension 95%+ 的時間都在等雲端 API 回傳翻譯結果，你的電腦幾乎不用做任何額外運算。翻譯過程中可以正常捲動、點擊、操作頁面。

**翻完不留痕跡。** 翻譯完成後，網頁上沒有被塞進多餘的 HTML 元素（零個 `shinkansen` class 的節點、Toast 已自動消失）。原地替換策略讓 DOM 節點數反而比翻譯前少 3,171 個（因為中文比英文精簡）。

**翻譯過的內容會自動快取。** 同一頁面再次開啟不用重新花錢。

以下是完整的技術測試數據。

## 記憶體佔用（JS Heap）

| 時間點 | JS Heap Used |
|--------|-------------|
| 翻譯前（頁面載入 + content script 注入後） | 10.33 MB |
| 翻譯完成後 | 18.24 MB |
| **增量** | **+7.91 MB** |

翻譯 989 段文字後，JS heap 增加約 7.9 MB。增量主要來自：

- **段落快取**（`STATE.cache`，Map）：儲存原文 → 譯文的對照，供重複段落直接命中。
- **原文備份**（`STATE.originalHTML`，Map）：保存每個被替換元素的原始 HTML，供「還原原文」功能使用。
- **Content Guard 快取**（`STATE.translatedHTML`，Map）：防止 SPA 框架覆寫譯文（v1.6+ 起新加，當時 v1.0.23 還沒有這條保護）。
- **Log buffer**（Service Worker 端，上限 1,000 條）。

翻譯完成後，頁面上沒有殘留任何 Shinkansen 的 DOM 元素（零個帶 `shinkansen` class 的節點、Toast 已自動消失）。Extension 不會在頁面上留下持久性的 DOM 痕跡。

## DOM 影響

| 指標 | 翻譯前 | 翻譯後 | 變化 |
|------|--------|--------|------|
| DOM 節點數 | 16,398 | 13,227 | **−3,171** |
| 注入的 style 元素 | 0 | 0 | — |
| 含 `shinkansen` class 的元素 | 0 | 0 | — |
| 標記 `data-shinkansen-translated` 的元素 | 0 | 959 | 不算節點數增加（只是在原 element 加屬性） |

DOM 節點數反而減少。Extension 採用「原地替換文字節點 + 在原 element 上加 `data-shinkansen-translated` 屬性」策略，不額外包裝 `<span>` 或 `<div>`，因此不會膨脹 DOM。`data-shinkansen-translated` 是用來標示「這個元素已翻譯」供 SPA Content Guard 比對的 attribute，不是新增節點。

### API 延遲分佈

| 指標 | 數值 |
|------|------|
| 平均回應時間 | 4,220 ms |
| 最快回應 | 863 ms |
| 最慢回應 | 14,940 ms |

翻譯期間超過 95% 的時間花在等待 API 回應。Content script 與 Service Worker 在等待期間幾乎是閒置的，不佔用 CPU 資源。

本地端 CPU 佔用集中在兩個短暫時間點：

1. **翻譯啟動時**：`collectParagraphs()` 遍歷整個 DOM tree 進行段落偵測，屬一次性同步操作。
2. **每批翻譯回來時**：DOM 注入（替換文字節點），每次耗時極短。

## 與 v1.0.23（2026-04-11）測試結果差異

> 同條件、同頁面、同模型。下表「Δ」為 v1.9.16 相對 v1.0.23 的變化。

### 成本與效率

| 指標 | v1.0.23 | v1.9.16 | Δ |
|------|---------|---------|---|
| 總費用 USD | $0.078 | $0.077 | **基本持平**（−$0.001） |
| 每段平均費用 USD | $0.000078 | $0.0000777 | 持平 |
| 批次總數 | 80 | 97 | +17（同段數但分批策略略微不同） |
| API 呼叫總數 | 147 | 97 | **−50**（v1.0.23 含 mismatch retry 重發，現行 batch:call=1:1） |
| 翻譯段落數 | 1,000 | 989 | −11（Wikipedia Taiwan 條目段落數與當時略有差異） |
| 翻譯後文字量 | 97,955 字 | 106,444 字 | +8,489（同一條目近一年內擴充） |

### 延遲

| 指標 | v1.0.23 | v1.9.16 | Δ |
|------|---------|---------|---|
| 總翻譯時間（壁鐘） | ~40 秒 | 48.4 秒 | **+8.4 秒**（批次多 17 批 + tail latency 拉長） |
| 平均 API 回應 | 5,125 ms | 4,220 ms | **−905 ms**（快 18%） |
| 最快回應 | 1,263 ms | 863 ms | **−400 ms**（快 32%） |
| 最慢回應 | 10,448 ms | 14,940 ms | **+4,492 ms**（tail 變慢 43%） |

### DOM 影響

| 指標 | v1.0.23 | v1.9.16 | Δ |
|------|---------|---------|---|
| DOM 翻譯前 | 16,286 | 16,398 | +112（Wikipedia 條目本身節點略增） |
| DOM 翻譯後 | 13,109 | 13,227 | +118（同上） |
| DOM 淨變化 | −3,177 | −3,171 | **持平**（同樣減少約 3,170 節點） |
| 含 `shinkansen` class 節點 | 0 | 0 | 持平 |
| `data-shinkansen-translated` 標記節點 | 未量測 | 959 | 新增屬性，不增節點數 |

### 記憶體

| 指標 | v1.0.23 | v1.9.16 | Δ |
|------|---------|---------|---|
| JS Heap 翻譯前 | 40.72 MB | 10.33 MB | **−30.4 MB**（baseline 差異，見備註） |
| JS Heap 翻譯後 | 35.34 MB | 18.24 MB | **−17.1 MB** |
| JS Heap 增量 | **−5.38 MB** | **+7.91 MB** | **+13.3 MB**（翻譯後 heap 不再下降反而上升） |

> Heap baseline 差異備註：v1.0.23 報告未說明測試方法是否含使用者捲動 / 互動暖機，v1.9.16 是 Playwright headless 冷啟動後立即量測，兩者 baseline 不可直接比較。**真正可比較的是「增量」**——v1.9.16 翻譯後 heap 增加 7.9 MB，比 v1.0.23 多出 13.3 MB。
>
> 增量變大的可能來源：(1) v1.6+ 加入的 Content Guard 結構（`STATE.translatedHTML`），(2) v1.6.11 起的 SPA observer，(3) v1.5.0 dual mode 基礎設施常駐，(4) v1.6.6 起的 sticky tabs / preset 系統。即便如此，絕對值（18 MB）仍遠低於頁面整體 heap quota（4 GB），對使用者無實質影響。

### 結構性差異（功能演進對效能的影響）

v1.0.23 → v1.9.16 這一年間加了以下功能，但**沒有讓「網頁翻譯」這條路徑變得更貴或更慢**：

- v1.5.0 雙語對照模式（本次測試走 single 不啟用）
- v1.5.6 中國用語黑名單（注入 prompt，本次測試清單空）
- v1.6+ SPA Content Guard（防止 React/Vue re-render 覆寫譯文）
- v1.6+ 翻譯快取版本遷移、跨網域 sticky tabs
- v1.7+ 術語表抽取 / 一致化（本次未啟用）
- v1.8.x YouTube 字幕、PDF 文件翻譯、ASR 字幕翻譯（不在本次路徑上）
- v1.8.59+ 多語目標（zh-TW / zh-CN / en / ja / ko / es / fr / de）
- v1.9.x Safari macOS 版打包、iOS 版規畫

新增的功能多數是條件性啟動（dual mode、術語表、字幕等），不啟用時不進 prompt 也不耗 token，本次「網頁翻譯 Wikipedia」路徑的成本曲線維持原狀。

### 一句話結論

> **「網頁翻譯」核心路徑在一年內加了大量功能，但成本、DOM 影響、CPU 佔用幾乎沒變動；唯一明顯增加的是常駐 JS heap（+13 MB），對使用者無感。**

## 備註

- 本測試在 extension 翻譯快取清空（worst case）的狀態下進行；實際使用時快取命中率會顯著降低 API 呼叫次數與總耗時。
- 量測工具：`tools/probe-performance.js`（v1.9.16 新增）。原始輸出存於 `.playwright-mcp/performance-result.json`。
- Wikipedia Taiwan 是刻意選擇的壓力測試頁面（近 1,000 段），一般網頁的段落數遠低於此。
- 翻譯速度主要受 API 延遲與並行批次數限制，與本地效能無關。提高 `maxConcurrent` 可縮短總耗時，但會更快消耗 rate limit 配額。
- 記憶體數據使用 `performance.memory` API（Chrome 限定）量測 JS heap，不含 V8 外部記憶體（如 DOM 本身的 C++ 物件），但差異通常很小。
- Gemini implicit context cache（cached_content_tokens）在首次冷啟跑為 0；後續同一 SW session 重跑會逐漸命中，實際使用情境通常會有一定折扣。
